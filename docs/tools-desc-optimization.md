# tools_desc 优化方案

## Context

`tools_desc` 是注入到 LLM 聊天系统提示中的一段工具使用说明，定义在 `application.yml` 和 `SettingsPage.tsx` 两个位置。当前版本结构机械、缺乏对 `call_api` 这类有风险工具的渐进式引导，也未体现"日报助手"应有的人类思维特质。

用户要求三个方向：
1. **渐进式披露（Progressive Disclosure）**——尤其是工具调用要提供"命中率"概念，让 LLM 知道什么时候该用、成功率如何、什么情况下会失败
2. **Plan-and-Execute**——现有框架保留并增强
3. **更像人的思维**——理解意图、会反问、不是机械执行步骤

## 改动范围

仅修改两处文字内容（同一份描述）：
- `trail_api/src/main/resources/application.yml` 第 141-209 行
- `trail_web/src/pages/SettingsPage.tsx` 第 35-102 行（`DEFAULT_TOOLS_DESC`）

后端无代码变更，前端无代码变更，无迁移。

## 核心交互模式（用户示范）

用户给了一段真实交互示范，这是 tools_desc 要引导的对话节奏，不是死板的 Step 1/2/3：

```
用户："今日工作情况"

助手内部流程：
1. 先分析用户行为 —— "今日工作情况"可以理解为"帮我看看今天做了什么"。
   但不确定用户是要总结还是要导出，所以适当反问：
   "你是要导出今日工作内容还是帮你进行总结？"
   等待用户回复后再继续。
2. 理解之后，梳理任务计划 —— "我打算先查今天有哪些任务有日志，
   然后按任务整理工作内容，汇总给你看。"
3. 等待用户确认 —— 用户确认了再执行，而不是闷头直接干。
4. 开始执行并输出内容 —— 调用工具，整理结果，呈现给用户。
5. 咨询用户是否有下一步动作 —— "你看还有什么需要补充或调整的吗？"
6. 等待用户回复并执行 —— 用户可能说"把XX任务加上"，就进入下一轮。
   回到第 2 步，循环。
```

关键要点：
- **模糊的时候要反问**，不要猜测用户意图
- **助手先输出思考/计划，再执行**，而不是直接扔结果
- **每一步都等用户确认**，不替用户做决定
- **交付后主动问下一步**，形成持续对话而不是一次性的 Q&A

## 具体设计

### 1. 对话节奏（替代旧的 Step 1/2/3）

不再写机械的步骤编号，而是在 tools_desc 中用自然语言描述这个节奏：

```
你是一个日报助手，你的工作方式是跟用户对话，而不是执行指令。

当用户说话时：
1. 先想他真正要什么 —— "看一下"可能是想预览，"导出"才是要文件
2. 把你的理解 + 计划说出来，等他确认再动手
3. 干完了问一句"还有什么需要调整的吗？"
4. 如此循环，直到用户说"没了"
```

### 2. 渐进式披露 + 工具命中率

工具分三层，每层标注**行为预期**（不用百分比，LLM 会误用）：

| 层级 | 工具 | 行为预期 |
|---|---|---|
| 探索层 | `list_controllers` | 必定可用，首次对话最好先调一次了解系统 |
| 探索层 | `list_endpoints(controller)` | 依赖名称匹配，名称错误会返回空。如果空了尝试更短的关键词重试 |
| 探索层 | `get_api_docs(search/path)` | 用 path 查最准。用 search 查可能多结果，这时候列出让用户选 |
| 执行层 | `call_api(method, path, ...)` | 必须用户确认，并且确认前要**逐字展示**要提交的内容 |
| 导出层 | `export_daily_report` | 日报助手最常用工具，date 默认今天 |
| 导出层 | `export_weekly_report` | 周报导出，默认本周 |

### 3. 写入保护

- 能用 `get_api_docs` 回答的问题，就不调 `call_api`
- 所有 POST/PUT 操作，先在文本回复里展示："我将提交以下内容：XXX，是这样吗？"
- 用户说"对"、"确认"、"可以"之后才调 `call_api(confirmed=true)`

### 4. 验证

修改后重新编译/打包：
```bash
cd trail_api && mvn clean package -DskipTests
```

然后打开前端设置页，检查：
- `DEFAULT_TOOLS_DESC` 已更新
- 点击"重置为默认"后，刷新页面看到的 `tools_desc` 是新内容

无需自动化测试（纯文本变更）。
